# lesson08: テストレベル — コンポーネントから受け入れまでの5つの段階

## このレッスンで学ぶこと

- テストレベルの定義と、SDLC（ソフトウェア開発ライフサイクル）との関係を理解する
- 5つのテストレベルそれぞれの目的とテスト対象を区別できるようになる（K2）
- 各テストレベルの主な担当とテストベースの違いを整理する
- 受け入れテストの主な形態を識別できるようになる
- テストレベルとテストタイプが異なる概念であることを説明できるようになる

## テストレベルの定義

**テストレベル**とは、系統的にまとめてマネジメントしていく、テスト活動のグループです。

各テストレベルは、テストプロセス（[lesson04](/lessons/lesson04/)）のインスタンスとして実施されます。対応する開発段階は、個々のコンポーネント単位から完成したシステムまで、必要に応じてシステムオブシステムズまで広がります。

### SDLCとの関係

テストレベルは、SDLC内の他の活動と関連づけられます（SDLCモデルの詳細は [lesson06](/lessons/lesson06/)）。押さえておきたいのは次の4点です。

- シーケンシャルなSDLCモデルでは、あるテストレベルの終了基準が次のテストレベルの開始基準の一部になるように定義されることが多い（開始基準と終了基準は [lesson23](/lessons/lesson23/)）
- イテレーティブなモデルの中には、この考え方が当てはまらないものもある
- 開発活動が複数のテストレベルにまたがることがある
- テストレベルどうしが時間的に重なることもある

## 5つのテストレベル

シラバスは次の5つのテストレベルを説明しています。図の左から右へ、テスト対象が小さな単位から大きな単位へと広がります。ただし前述の通り、テストレベルは時間的に重なることもあり、常に一方通行で進むわけではありません。

```mermaid
flowchart LR
    A[コンポーネントテスト\n単独のコンポーネント] --> B[コンポーネント統合テスト\nコンポーネント間の\nインターフェース] --> C[システムテスト\nシステム全体の\n振る舞いと能力] --> D[システム統合テスト\n他システムや\n外部サービスとの連携] --> E[受け入れテスト\n妥当性確認と\nデプロイ準備の実証]
```

まず全体を表で整理します。

| テストレベル | 主なテスト対象 | 主な目的 |
|------|------|------|
| コンポーネントテスト | 単独のコンポーネント | コンポーネントが単独で正しく動作することを確認する |
| コンポーネント統合テスト | コンポーネント間のインターフェースと相互処理 | 組み合わせたコンポーネントが正しく連携することを確認する |
| システムテスト | システムやプロダクト全体 | システム全体の振る舞いや能力を評価する |
| システム統合テスト | テスト対象システムと他のシステムや外部サービスとのインターフェース | 外部との連携が正しく機能することを確認する |
| 受け入れテスト | システム全体 | 妥当性確認と、デプロイの準備ができていることを実証する |

続いて、1つずつ特徴を見ます。

### コンポーネントテスト

ユニットテストとも呼ばれます。コンポーネントを単独でテストすることに焦点を当てます。

- テストハーネスやユニットテストフレームワークのような、特定のサポートを必要とすることが多い
- 通常、開発担当者が開発環境で実施する

::: info テストハーネスとは
テスト対象のコンポーネントを単独で動かすために、スタブやドライバーなどを組み合わせて用意するテスト環境です。呼び出し元や呼び出し先がまだ存在しなくても、コンポーネント単体の動作を確認できます。
:::

### コンポーネント統合テスト

ユニット統合テストとも呼ばれます。コンポーネント間のインターフェースと相互処理に焦点を当てます。

このテストレベルは、統合戦略のアプローチに大きく依存します。シラバスが挙げる統合戦略は次の3つです。

| 統合戦略 | 進め方 |
|------|------|
| ボトムアップ | 下位のコンポーネントから順に統合する |
| トップダウン | 上位のコンポーネントから順に統合する |
| ビッグバン | すべてのコンポーネントを一度に統合する |

### システムテスト

システムやプロダクト全体の振る舞いや能力の全般に焦点を当てます。多くの場合、エンドツーエンドのタスクに対する機能テストや、品質特性に対する非機能テストを行います（機能テストと非機能テストは [lesson09](/lessons/lesson09/)）。

- 使用性テストなど一部の非機能品質特性は、本番相当のテスト環境で、すべて揃ったシステムを対象にテストすることが望ましい
- サブシステムのシミュレーションも使用できる
- システムの仕様に関連するテストであり、独立したテストチームが実施することがある（テストの独立性は [lesson05](/lessons/lesson05/)）

### システム統合テスト

テスト対象のシステムと、他のシステムや外部サービスとのインターフェースのテストに焦点を当てます。できれば運用環境に近い、適切なテスト環境が必要になります。

::: tip 2つの統合テストの区別
コンポーネント統合テストは1つのシステムの内部でのコンポーネントどうしの統合、システム統合テストはシステムとその外部にある他システムや外部サービスとの統合を対象にします。「統合」という言葉だけで判断せず、何と何をつなぐのかで区別しましょう。
:::

### 受け入れテスト

妥当性確認（[lesson01](/lessons/lesson01/)）と、デプロイの準備ができていることの実証に焦点を当てます。これは、システムがユーザーのビジネスニーズを満たしていることを意味します。想定ユーザーが実施することが理想とされています。

#### 受け入れテストの主な形態

受け入れテストには、目的や実施環境の異なる複数の形態があります。

| 形態 | 焦点 |
|------|------|
| ユーザー受け入れテスト（UAT） | 想定ユーザーが、システムでビジネスニーズを満たせるかを確認する |
| 運用受け入れテスト | 運用担当者やシステム管理者が、バックアップとリストアなどの運用面を確認する |
| 契約による受け入れテスト | 契約で合意した受け入れ基準を満たしているかを確認する |
| 規制による受け入れテスト | 法律や規制に適合しているかを確認する |
| アルファテスト | 開発組織側の環境で、想定ユーザーなどが実施する |
| ベータテスト | ユーザー自身の環境で、想定ユーザーが実施する |

## テストレベルを区別する属性

テストレベルは、テスト活動の重複を避けるために区別します。シラバスが挙げる代表的な属性は次の5つです。

- テスト対象
- テスト目的
- テストベース
- 欠陥および故障
- アプローチと責務

このうち、担当とテストベース（[lesson04](/lessons/lesson04/)）の代表的な違いを整理すると次のようになります。

| テストレベル | 主な担当 | 代表的なテストベース |
|------|------|------|
| コンポーネントテスト | 開発担当者 | コード、詳細設計 |
| コンポーネント統合テスト | 通常は開発担当者 | インターフェース仕様、アーキテクチャー設計 |
| システムテスト | 独立したテストチームが実施することがある | システムの仕様、要件 |
| システム統合テスト | テストチーム | 外部システムとのインターフェース仕様 |
| 受け入れテスト | 想定ユーザーが理想 | ビジネスニーズ、契約や規制の要件 |

どのテストレベルでも同じことを重複して確認するのではなく、レベルごとに対象と目的を変えることで、テスト全体の効率を高めます。

## テストレベルとテストタイプの違い

テストレベルと混同しやすい概念に、テストタイプがあります（[lesson09](/lessons/lesson09/)）。両者は異なる軸でテスト活動をグループ化したものです。

| 観点 | テストレベル | テストタイプ |
|------|------|------|
| 定義 | 系統的にまとめてマネジメントしていくテスト活動のグループ | 特定の品質特性に関連するテスト活動のグループ |
| 対応する軸 | 開発段階（コンポーネントからシステム全体まで） | 品質特性（機能、性能効率性、使用性など） |
| 例 | コンポーネントテスト、システムテスト、受け入れテストなど | 機能テスト、非機能テストなど |

テストタイプに属するテスト活動のほとんどは、すべてのテストレベルで実行できます。たとえば機能テストというテストタイプは、コンポーネントテストでもシステムテストでも受け入れテストでも実施できます。

::: warning レベルとタイプは直交する概念
「システムテストだから機能テスト」のような1対1の対応はありません。レベルは「いつ、何を対象に」、タイプは「どの品質特性に着目して」テストするかを表します。試験でも両者の混同を突く選択肢が出やすいので、軸の違いを意識して区別しましょう。
:::

## キーワード

| 用語 | 説明 |
|------|------|
| テストレベル（test level） | 系統的にまとめてマネジメントしていくテスト活動のグループ。開発段階に応じたテストプロセスのインスタンス |
| コンポーネントテスト（component testing） | コンポーネントを単独でテストすることに焦点を当てるテストレベル。ユニットテストとも呼ぶ |
| コンポーネント統合テスト（component integration testing） | コンポーネント間のインターフェースと相互処理に焦点を当てるテストレベル。ユニット統合テストとも呼ぶ |
| システムテスト（system testing） | システムやプロダクト全体の振る舞いや能力の全般に焦点を当てるテストレベル |
| システム統合テスト（system integration testing） | テスト対象システムと他のシステムや外部サービスとのインターフェースに焦点を当てるテストレベル |
| 受け入れテスト（acceptance testing） | 妥当性確認と、デプロイの準備ができていることの実証に焦点を当てるテストレベル |

## 試験のポイント

- 5つのテストレベルの名前と焦点を対応づけられるようにする（K2で区別が問われる）
- 各レベルの主な担当を対応づける（コンポーネントテストとコンポーネント統合テストは開発担当者、システムテストは独立したテストチームが実施することがある、受け入れテストは想定ユーザーが理想）
- コンポーネント統合テスト（システム内部のコンポーネント間）とシステム統合テスト（他のシステムや外部サービスとの間）は「何と何をつなぐか」で区別する
- 受け入れテストは妥当性確認とデプロイ準備の実証に焦点を当て、主な形態（UAT・運用・契約による・規制による・アルファ・ベータ）を識別できるようにする
- シーケンシャルなSDLCモデルでは、あるテストレベルの終了基準が次のテストレベルの開始基準の一部になることが多い（イテレーティブなモデルには当てはまらないものもある）
- テストレベルは開発段階に、テストタイプは品質特性に対応し、テストタイプのテスト活動のほとんどはすべてのテストレベルで実行できる
