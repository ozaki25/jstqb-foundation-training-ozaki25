# lesson20: 経験ベースのテスト技法 — 知識と経験を欠陥の発見に活かす

## このレッスンで学ぶこと

- エラー推測がテスト担当者のどのような知識に基づいて欠陥を予測するかを説明できるようになる（K2）
- フォールト攻撃によるエラー推測の系統的な進め方を理解する
- 探索的テストの特徴と、セッションベースドテストの流れを説明できるようになる（K2）
- チェックリストベースドテストの特徴と、チェックリストを更新し続ける理由を説明できるようになる（K2）
- 経験ベースのテスト技法がブラックボックス・ホワイトボックスの技法を補完する位置づけを押さえる

## 経験ベースのテスト技法の位置づけ

経験ベースのテスト技法は、テスト担当者の知識と経験からテストを導出するカテゴリーです（テスト技法の3カテゴリーは [lesson14](/lessons/lesson14/)）。仕様や内部構造といった文書化された情報ではなく、「どこで欠陥が起きやすいか」を知っている人の頭の中を活用します。

シラバスが取り上げる技法は次の3つです。

| 技法 | 考え方 | 使いどころ |
|------|------|------|
| エラー推測 | 過去の欠陥や開発担当者が起こしやすいエラーの知識から、欠陥の存在を予測してテストを設計する | 欠陥や故障のデータ・経験が蓄積されている領域を効率よく攻めたいとき |
| 探索的テスト | テスト対象について学習しながら、テストケースの設計・実行・評価を同時に行う | 仕様が不十分なときや、テストのスケジュールに余裕がないとき |
| チェックリストベースドテスト | 経験などから作ったチェックリストのテスト条件をカバーするようにテストする | 詳細なテストケースがない状況で、ガイドラインと一貫性がほしいとき |

3つの技法に共通する性質も押さえておきましょう。

- 技法の有効性は、テスト担当者のスキル（経験・ドメイン知識・分析力など）に大きく依存する
- ブラックボックステスト技法やホワイトボックステスト技法では見逃すような欠陥も検出できる
- したがって、ほかのカテゴリーの技法を置き換えるものではなく、補完するものである

## エラー推測

エラー推測は、エラー・欠陥・故障の発生をテスト担当者の知識に基づいて予測する技法です。「このあたりが怪しい」という勘に見えますが、その裏付けは具体的な知識です。

### 予測のもとになる知識

シラバスは、予測のもとになる知識として次の3つを挙げています。

- アプリケーションが過去にどのように動作してきたか
- 開発担当者が起こしやすいエラーの種類と、そのエラーに起因する欠陥の種類
- 他の類似アプリケーションで発生した故障の種類

エラー・欠陥・故障の区別は [lesson02](/lessons/lesson02/) で学んだ通りです。「人のエラーが欠陥を生み、欠陥が故障を引き起こす」という連鎖のどこに注目してもかまいません。

### 欠陥が現れやすい領域

一般に、エラー・欠陥・故障は次のような領域に関連して発生します。エラー推測では、こうした「定番の落とし穴」を狙います。

| 領域 | 例 |
|------|------|
| 入力 | 正しい入力を受け付けない、パラメーターの誤りや不足 |
| 出力 | 誤った形式、誤った結果 |
| 論理 | 条件の不足、誤った演算子 |
| 計算 | 誤ったオペランド、誤った計算 |
| インターフェース | パラメーターの不一致、互換性のない型 |
| データ | 誤った初期化、誤った型 |

### フォールト攻撃

**フォールト攻撃**は、エラー推測を系統的に実施するためのアプローチです。個人の勘に任せるのではなく、リストを使って計画的に攻めます。

進め方は次の通りです。

1. 想定されるエラー・欠陥・故障のリストを作成する（または既存のリストを入手する）
2. リストの項目を狙うテストケースを設計する（エラーに関連する欠陥の特定、欠陥の顕在化、故障の誘発）

リストのもとになるのは、経験、欠陥や故障のデータ、そしてソフトウェアが不合格になる理由についての一般的な知識です。

## 探索的テスト

探索的テストでは、テスト担当者がテスト対象について学習しながら、テストケースの設計・実行・評価を同時に行います。「事前にすべて設計してから実行する」という進め方とは対照的です。

探索的テストの使い道として、シラバスは次の3つを挙げています。

- テスト対象についてより深く知る
- 注力したテストでより深く探索する
- まだテストしていない領域のテストを作成する

### セッションベースドテスト

探索的テストは自由度が高いぶん、無計画に流れやすい技法でもあります。活動を体系的にする方法として、**セッションベースドテスト**があります。

```mermaid
flowchart LR
    A[テストチャーター\nテスト目的を定義] --> B[テストセッション\n決められた時間枠内で\n設計・実行・評価を同時進行] --> C[デブリーフィング\nステークホルダーと\nテスト結果を議論]
```

セッションベースドアプローチのポイントは次の通りです。

- 探索的テストを、あらかじめ決められた時間枠内で行う
- テスト担当者は、テスト目的を含む**テストチャーター**に従ってテストを実行する（テストチャーターはテスト設計の作業成果物。[lesson04](/lessons/lesson04/)）
- テストセッションの後には通常、テスト担当者と、テスト結果に関心を持つステークホルダーとの間で議論する**デブリーフィング**を行う

カバレッジとの関係も整理されています。テスト目的はハイレベルのテスト条件として扱われることがあり、カバレッジアイテムはテストセッション中に識別して確認します。また、テスト担当者は**テストセッションシート**を使って、実行した手順や発見した事象を文書化する場合があります。

### 探索的テストの使いどころ

探索的テストが役立つのは、次のような状況です。

- 仕様がほとんどない、または不十分である
- テストのスケジュールに余裕がない
- 他の形式的なテスト技法を補完したい

探索的テストは他のテスト技法と併用できます。たとえば、探索の途中で入力の同値パーティションに気づいたら、同値分割法（[lesson15](/lessons/lesson15/)）の考え方で入力を選べます。また、限られた時間で何をどこまで探索するかの判断には、リスク分析の結果が手がかりになります（リスクベースドテストは [lesson25](/lessons/lesson25/)）。

::: tip 効果はテスト担当者のスキル次第
探索的テストは、テスト担当者が経験豊富でドメインの知識を持ち、分析力・好奇心・創造性といったスキルを高いレベルで備えているほど効果的になります。こうした汎用スキルは [lesson05](/lessons/lesson05/) で扱っています。
:::

## チェックリストベースドテスト

チェックリストベースドテストでは、チェックリストに挙げたテスト条件をカバーするように、テストケースを設計・実装・実行します。

### チェックリストの作り方

チェックリストのもとになるのは、次のような知識です。

- 経験
- ユーザーにとって何が重要かという知識
- ソフトウェアが不合格となる理由と仕組みについての理解

項目の書き方にも指針があります。

- 項目は多くの場合、質問の形で表現する
- 各項目は、個別に直接チェックできるようにする
- 項目は、要件・GUI の設定・品質特性・その他のテスト条件を参照することがある

チェックリストは、機能テストにも非機能テストにも使えます。非機能の例としては、使用性テストのための10のヒューリスティックがよく知られています。

::: warning チェックリストに含めない方がよい項目
自動的にチェックできる項目、開始基準・終了基準（[lesson23](/lessons/lesson23/)）として扱う方が適した項目、一般的すぎる項目は、チェックリストに含めない方がよいとされています。
:::

### チェックリストの更新

チェックリストは、作って終わりではありません。時間の経過とともに手入れが必要です。

- 開発担当者は同じエラーを避けることを学ぶため、追加した項目の効果は時間とともに薄れていくことがある
- 新たに発見した重要度の高い欠陥を反映するために、新しい項目の追加が必要になる場合もある

したがって、チェックリストは欠陥分析に基づいて定期的に更新すべきです。ただし、項目を足し続けてチェックリストが長くなりすぎないように注意します。

### 一貫性とカバレッジのトレードオフ

詳細なテストケースがない場合でも、チェックリストベースドテストはテストのガイドラインと、ある程度の一貫性を与えてくれます。

一方、チェックリストがハイレベルである場合は、次のトレードオフが生じます。

- 実際のテストにある程度のばらつきが生じ、その結果カバレッジが大きくなる可能性がある
- ただし、再現性は低くなる

## キーワード

| 用語 | 説明 |
|------|------|
| エラー推測（error guessing） | エラー・欠陥・故障の発生を、テスト担当者の知識に基づいて予測する技法 |
| フォールト攻撃（fault attack） | エラー推測を系統的に実施するアプローチ。想定されるエラー・欠陥・故障のリストを作成または入手し、それらを狙うテストケースを設計する |
| 探索的テスト（exploratory testing） | テスト担当者がテスト対象について学習しながら、テストケースを同時に設計・実行・評価するテスト |
| セッションベースドテスト（session-based testing） | 探索的テストを体系的に行うアプローチ。あらかじめ決められた時間枠内で、テストチャーターに従ってテストする |
| テストチャーター（test charter） | テストセッションで達成すべきテスト目的を記述したもの。テスト設計の作業成果物の1つ |
| デブリーフィング（debriefing） | テストセッションの後に、テスト担当者とテスト結果に関心を持つステークホルダーとの間で行う議論 |
| テストセッションシート（test session sheet） | テストセッション中に実行した手順や発見した事象を文書化するためのシート |
| チェックリストベースドテスト（checklist-based testing） | チェックリストに挙げたテスト条件をカバーするように、テストケースを設計・実装・実行する技法 |

## 試験のポイント

- 4.4.1〜4.4.3はK2（説明する）のため、3つの技法それぞれの進め方と使いどころを説明できるようにする
- エラー推測は「過去の動作状況」「開発担当者が起こしやすいエラーとそれに起因する欠陥」「類似アプリケーションの故障」の3つの知識に基づいて欠陥を予測する
- エラー・欠陥・故障は入力・出力・論理・計算・インターフェース・データに関連して発生することが多い
- フォールト攻撃は、想定されるエラー・欠陥・故障のリストを作成または入手してテストケースを設計する、エラー推測の系統的なアプローチ
- 探索的テストの軸は「同時に」で、テスト対象について学習しながらテストケースの設計・実行・評価を同時に行う（事前にすべて設計してから実行する進め方との対比がひっかけどころ）
- セッションベースドテストは、決められた時間枠内でテスト目的を含むテストチャーターに従って実行し、終了後にデブリーフィングを行う
- テストチャーター（テスト目的を定義する）とテストセッションシート（実行した手順や発見した事象を文書化する）の役割の取り違えに注意
- 探索的テストは、仕様が不十分な場合やスケジュールに余裕がない場合に役立ち、他の形式的なテスト技法を補完する場合にも効果が大きい
- チェックリストの項目は経験・ユーザーにとって重要なこと・ソフトウェアが不合格となる理由の理解から作り、欠陥分析に基づいて定期的に更新する（開発担当者が同じエラーを避けることを学ぶため、項目の効果は薄れていく）
- ハイレベルなチェックリストはカバレッジが大きくなる可能性がある一方、再現性は低くなるというトレードオフを押さえる
- 経験ベースの技法の有効性はテスト担当者のスキルに大きく依存し、ブラックボックス・ホワイトボックスのテスト技法を置き換えるのではなく補完する
