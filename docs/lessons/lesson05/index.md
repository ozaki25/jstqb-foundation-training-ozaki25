# lesson05: テストに必要なスキルとチーム — 汎用的スキル・チーム全体アプローチ・テストの独立性

## このレッスンで学ぶこと

- テストを行うために求められる汎用的スキルの例を挙げられるようになる
- 欠陥や故障の情報を建設的に伝えることが重要な理由を理解する
- チーム全体アプローチの考え方と利点を想起できるようになる
- テストの独立性の度合いを段階として説明できるようになる
- テストの独立性の長所と短所を区別できるようになる

## テスト担当者に求められる汎用的スキル

スキルとは、知識・実践・適性に基づいて何かをうまく行う能力のことです。よいテスト担当者は、自分の仕事をうまくこなすためのスキルを備え、効果的なチームプレーヤーとして働き、さまざまな独立性の度合いでテストを実行できることが求められます。

汎用的でありながら、テスト担当者に特に関連が深いスキルとして、シラバスは次の例を挙げています。

| スキル | テストにおける役割 |
|------|------|
| テストの知識 | テスト技法の活用などにより、テストの**有効性**を高める |
| 徹底さ・慎重さ・好奇心・細部へのこだわり・理路整然さ | 発見しにくい欠陥を識別する |
| コミュニケーションスキル・積極的な傾聴・チームプレーヤーであること | ステークホルダーと効果的にやりとりし、情報を伝えて理解してもらい、欠陥を報告して議論する |
| 分析的思考・批判的思考・創造性 | テストの**有効性**を高める |
| 技術的な知識 | 適切なテストツールの使用により、テストの**効率性**を高める |
| ドメイン知識 | エンドユーザーやビジネス側代表者を理解し、コミュニケーションできるようになる |

::: tip 有効性と効率性の対応
テストの知識や分析的思考・批判的思考・創造性が高めるのは「有効性（どれだけ効果があるか）」、技術的な知識とテストツールが高めるのは「効率性（どれだけ少ないコストで達成できるか）」です。どのスキルが何を高めるかの対応は、選択肢の入れ替えで問われやすいポイントです。
:::

### 欠陥や故障の伝え方

テスト担当者は、欠陥や故障という「悪い知らせ」を届ける役回りになりがちです。悪い知らせの運び手を責めてしまうのは、人間の一般的な特性です。

伝え方が難しくなる背景として、次の点があります。

- テスト結果を伝えると、プロダクトやその作成者への批判と受け取られる可能性がある
- **確証バイアス**が働くと、現在持っている信念と異なる情報を受け入れにくくなる
- テストはプロジェクトの成功やプロダクトの品質に大きく貢献しているのに、破壊的な活動だと認識する人もいる

こうした見方を改善するために、欠陥や故障に関する情報は**建設的な方法で伝える**べきです。だからこそ、テスト担当者にとってコミュニケーションスキルは非常に重要です。

## チーム全体アプローチ

テスト担当者に重要なスキルの1つが、チームの中で効果的に働き、チームのゴールに積極的に貢献する能力です。この能力の上に成り立つのが**チーム全体アプローチ**（whole-team approach）で、アジャイル開発手法の1つであるエクストリームプログラミング（XP）に由来します（開発ライフサイクルとテストの関係は [lesson06](/lessons/lesson06/) で扱います）。

チーム全体アプローチの特徴は次の通りです。

- 必要な知識とスキルを持つチームメンバーであれば、誰がどのタスクを担当してもよい
- **チーム全員が品質に対して責任を持つ**
- 同じワークスペース（物理的または仮想的）を共有し、情報伝達と相互作用を活性化する

### チーム全体アプローチの利点

シラバスが挙げる利点は次の3つです。K1レベルの学習目標なので、そのまま想起できるようにしておきましょう。

- チームダイナミクスが向上する
- チーム内の情報伝達とコラボレーションが強化される
- チーム内のさまざまなスキルセットをプロジェクトの利益のために活用できる、相乗効果が生まれる

### テスト担当者と他のメンバーの協働

テスト担当者は、望ましい品質レベルの達成を確かなものにするため、他のチームメンバーと緊密に連携します。

| 協働の相手 | 協働の内容 |
|------|------|
| ビジネス側代表者 | 協力して適切な受け入れテストを作成する（協働的なテストアプローチは [lesson21](/lessons/lesson21/) で扱います） |
| 開発担当者 | 協働してテスト戦略に合意し、テスト自動化のアプローチを決定する |

こうした協働を通じて、テスト担当者はテストに関する知識を他のチームメンバーに広め、プロダクトの開発に影響を与えられます。

::: warning チーム全体アプローチが適さないコンテキスト
チーム全体アプローチが常に適切とは限りません。たとえばセーフティクリティカルな状況などでは、高いレベルのテストの独立性が必要になる場合があります。
:::

## テストの独立性

**テストの独立性**とは、テストを行う人が作業成果物の作成者からどれだけ離れているかという考え方です。ある程度の独立性があると、作成者とテスト担当者の認知バイアスの違いにより、テスト担当者がより効果的に欠陥を発見できるようになります。

ただし、独立性は開発担当者による慣れ親しんだ作業に取って代わるものではありません。開発担当者は、自分のコードにある多くの欠陥を効率的に発見できます。

### 独立性の度合い

誰が作業成果物をテストするかによって、独立性の度合いは段階的に変わります。

```mermaid
flowchart LR
    A["作成者自身\n独立性なし"] --> B["同じチームの仲間\nある程度の独立性あり"] --> C["チーム外・組織内の\nテスト担当者\n独立性が高い"] --> D["組織外の\nテスト担当者\n独立性がとても高い"]
```

| 誰がテストするか | 独立性の度合い |
|------|------|
| 作業成果物の作成者自身 | 独立性なし |
| 作成者と同じチームの仲間 | ある程度の独立性あり |
| 作成者のチーム外だが組織内のテスト担当者 | 独立性が高い |
| 組織外のテスト担当者 | 独立性がとても高い |

ほとんどのプロジェクトでは、複数の独立性のレベルを組み合わせてテストを実施するのが最善とされます。シラバスの例は次の通りです（テストレベルの詳細は [lesson08](/lessons/lesson08/) で扱います）。

| テスト活動 | 担当 |
|------|------|
| コンポーネントテスト、コンポーネント統合テスト | 開発担当者 |
| システムテスト、システム統合テスト | テストチーム |
| 受け入れテスト | ビジネス側代表者 |

### 独立性の長所と短所

独立性が高いほどよい、と単純には言えません。長所と短所の両方を区別して押さえましょう。

| 独立性の長所 | 独立性の短所 |
|------|------|
| 経歴・技術的視点・バイアスが異なるため、独立したテスト担当者は開発担当者と比較して異なる種類の故障や欠陥を認識する可能性が高い | 独立したテスト担当者が開発チームから孤立し、協調性の欠如・情報伝達の問題・開発チームとの敵対関係に陥る可能性がある |
| 仕様策定や実装の際にステークホルダーが設定した仮定を検証し、異議を唱え、反証できる | 開発担当者が品質に対する責任感を失うことがある |
| | 独立したテスト担当者がボトルネックとみなされたり、リリースの遅れを責められたりする可能性がある |

::: tip 長所と短所の見分け方
長所は「違う視点だから見つけられる・疑える」という発見力の話、短所は「距離があるから起きる」協調・責任感・スピードの問題です。この軸で整理すると、選択肢を長所と短所に振り分けやすくなります。
:::

## キーワード

| 用語 | 説明 |
|------|------|
| 確証バイアス（confirmation bias） | 現在持っている信念と異なる情報を受け入れにくくなる傾向。欠陥報告の受け止め方に影響する |
| チーム全体アプローチ（whole-team approach） | 必要な知識とスキルを持つメンバーなら誰がどのタスクを担当してもよく、全員が品質に責任を持つという考え方。XP（エクストリームプログラミング）に由来する |
| テストの独立性（independence of testing） | テストを行う人が作業成果物の作成者からどれだけ離れているかの度合い。認知バイアスの違いにより欠陥発見の効果を高める |

## 試験のポイント

- 汎用的スキルの例（テストの知識・細部へのこだわり・コミュニケーションスキル・分析的思考と批判的思考・技術的な知識・ドメイン知識など）を挙げられるようにする（K2）
- テストの知識や分析的思考・批判的思考・創造性は「有効性」を、技術的な知識とテストツールは「効率性」を高めるという対応を押さえる
- 欠陥や故障の情報を建設的に伝えるべき理由を押さえる（悪い知らせの運び手を責める人間の特性、確証バイアス、プロダクトや作成者への批判と受け取られる可能性）
- チーム全体アプローチはXPに由来し、利点3つ（チームダイナミクスの向上・情報伝達とコラボレーションの強化・スキルセットを活用する相乗効果）をK1で想起する（セーフティクリティカルな状況などでは常に適切とは限らない）
- 独立性の度合いは「作成者自身（なし）→ 同じチームの仲間（ある程度あり）→ 組織内のチーム外（高い）→ 組織外（とても高い）」の4段階で押さえる
- 独立性の長所（異なる故障や欠陥の認識・ステークホルダーの仮定への異議や反証）と短所（孤立・開発担当者の責任感の喪失・ボトルネック視）を区別する（K2）
