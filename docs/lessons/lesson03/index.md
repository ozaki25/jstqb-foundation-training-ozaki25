# lesson03: テストの7原則 — あらゆるテストに通じるガイドライン

## このレッスンで学ぶこと

- テストの7原則を漏れなく挙げられるようになる
- 各原則の意味を自分の言葉で説明できるようになる
- 全数テストの代わりにテストの労力を集中させる方法を理解する
- テストの弱化への対処と、繰り返しが有益になる例外を説明できるようになる
- 「欠陥ゼロ」の落とし穴と妥当性確認のつながりを理解する

## 7原則の一覧

長年にわたり、さまざまなテストの原則が提唱され、あらゆるテストに適用できる一般的なガイドラインとなってきました。シラバスはそのうち7つを示しています。

| # | 原則 | 要点 |
|---|------|------|
| 1 | テストは欠陥があることは示せるが、欠陥がないことは示せない | 欠陥が見つからなくても、テスト対象の正しさの証明にはならない |
| 2 | 全数テストは不可能 | すべてをテストする代わりに、労力を集中させる |
| 3 | 早期テストで時間とコストを節約 | 早い段階で欠陥を取り除くほど、品質コストが下がる |
| 4 | 欠陥の偏在 | 欠陥の大部分は少数のコンポーネントに集中する |
| 5 | テストの弱化 | 同じテストの繰り返しでは、新たな欠陥を見つける効果が薄れる |
| 6 | テストはコンテキスト次第 | 唯一普遍的に適用できるテストのアプローチは存在しない |
| 7 | 「欠陥ゼロ」の落とし穴 | 欠陥をすべて修正しても、ユーザーのニーズを満たすとは限らない |

7原則は暗記して終わりの知識ではなく、テストの計画や技法選択の判断に使う道具です。1つずつ確認していきます。

## 各原則の説明

### 原則1: 欠陥がないことは示せない

テストで欠陥を見つければ、テスト対象に欠陥があることを示せます。しかし、欠陥が見つからなかったとしても、それは欠陥がないことの証明にはなりません。

- テストによって、未検出のまま残る欠陥の数を減らせます
- それでも、テスト対象が正しいことは証明できません

実務では、「テストで問題が出なかった」を「欠陥ゼロの保証」と読み替えないことが大切です。

### 原則2: 全数テストは不可能

すべての入力や条件の組み合わせをテストすることは、ごく単純なソフトウェアを除いて非現実的です。たとえば入力欄が数個あるだけでも、値の組み合わせは爆発的に増えます。

全数テストの代わりに、次の手段でテストにかける労力を集中させます。

- テスト技法で入力に濃淡をつける（[lesson14](/lessons/lesson14/)）
- テストケースの優先順位付けで重要なものから実行する（[lesson24](/lessons/lesson24/)）
- リスクベースドテストでリスクの高い箇所に注力する（[lesson25](/lessons/lesson25/)）

### 原則3: 早期テストで時間とコストを節約

プロセスの早い段階で欠陥を取り除くと、その欠陥に起因する別の欠陥が後続の作業成果物に生まれることを防げます。SDLC（ソフトウェア開発ライフサイクル）の後半で発生する故障が減るため、品質コストを削減できます。

早い段階で欠陥を見つけるには、静的テスト（[lesson11](/lessons/lesson11/)）と動的テストの両方をなるべく早い時期に開始します。この考え方はシフトレフト（[lesson07](/lessons/lesson07/)）と呼ばれるアプローチの土台になっています。

### 原則4: 欠陥の偏在

見つかる欠陥や運用時の故障の大部分は、通常、少数のシステムコンポーネントに集中します。この現象は**パレートの法則**の一例です。

予測した欠陥の偏在と、テスト中や運用中に実際に観察した欠陥の偏在は、リスクベースドテスト（[lesson25](/lessons/lesson25/)）の重要なインプットになります。「欠陥が多く見つかったモジュールには、まだ欠陥が残っている可能性が高い」と考えて、テストの重点を決める材料にします。

### 原則5: テストの弱化

同じテストを何回も繰り返すと、新たな欠陥を検出する効果は薄れていきます。既存のテストで見つかる欠陥はすでに修正されており、残った欠陥は別の角度からでないと見つからないためです。

::: info 殺虫剤のパラドックス
この原則は、以前のシラバスでは「殺虫剤のパラドックス」と呼ばれていました。同じ殺虫剤を使い続けると害虫に効かなくなることへのたとえです。V4.0では「テストの弱化」という名前になりましたが、旧称で覚えている資料も多いので、同じ内容だと知っておくと混乱しません。
:::

この影響を克服するには、次の対処が必要になる場合があります。

- 既存のテストとテストデータを変更する
- 新規にテストを作成する

ただし例外もあります。自動化されたリグレッションテスト（[lesson09](/lessons/lesson09/)）のように、同じテストの繰り返しが「変更後もデグレードしていない」という有益な結果を示せる場合です。

### 原則6: テストはコンテキスト次第

テストに唯一普遍的に適用できるアプローチは存在しません。テストはコンテキストによって異なる方法で行われます。

たとえば、医療機器のような安全性が重要なシステムと、短いサイクルでリリースするECサイトでは、求められるテストの厳密さもドキュメントの量も異なります。「正しいテストのやり方」を1つに固定せず、状況に合わせて選ぶ姿勢が求められます。

### 原則7: 「欠陥ゼロ」の落とし穴

ソフトウェアを検証するだけでシステムを正しく構築できると期待することは、誤った思い込みです。指定された要件をすべて徹底的にテストし、検出した欠陥をすべて修正しても、次のようなシステムができあがることがあります。

- ユーザーのニーズや期待を満たさない
- 顧客のビジネスゴールの達成に役立たない
- 競合するシステムに比べて劣る

だからこそ、検証に加えて妥当性確認も実施すべきです。検証と妥当性確認の違いは [lesson01](/lessons/lesson01/) で扱っています。

::: tip 原則1と原則7の関係
原則1は「欠陥が見つからなくても正しさは証明できない」、原則7は「欠陥をすべて直しても役に立つとは限らない」という主張です。どちらも「テストを通ったから大丈夫」という過信を戒めていますが、原則1は欠陥の有無、原則7はニーズへの適合という別の観点を突いています。
:::

## キーワード

| 用語 | 説明 |
|------|------|
| 全数テスト | すべての入力や条件の組み合わせをテストすること。ごく単純なソフトウェア以外では非現実的 |
| 早期テスト | 静的テストと動的テストをなるべく早い時期に開始すること。品質コストの削減につながる |
| 欠陥の偏在 | 欠陥や運用時の故障の大部分が少数のシステムコンポーネントに集中する現象 |
| パレートの法則 | 結果の大部分が少数の原因から生じるという経験則。欠陥の偏在はその一例 |
| テストの弱化 | 同じテストの繰り返しで新たな欠陥の検出効果が薄れること。旧称は「殺虫剤のパラドックス」 |
| 「欠陥ゼロ」の落とし穴 | 検出した欠陥をすべて修正しても、ニーズを満たす有用なシステムになるとは限らないこと |
| リスクベースドテスト | リスクの分析結果に基づいてテストの労力を配分するアプローチ（詳細は [lesson25](/lessons/lesson25/)） |

## 試験のポイント

- 7原則は定番の出題テーマなので、7つすべてを名前つきで言えるようにしておく（原則名と説明文の組み合わせを選ぶ形式が多い）
- 原則1は「欠陥があることは示せる」と「欠陥がないことは示せない」の両面をセットで押さえる
- 原則2の対策として、テスト技法・テストケースの優先順位付け・リスクベースドテストの3つが挙げられている
- 原則3の根拠は「後続の作業成果物への欠陥の波及を防ぎ、SDLC後半の故障を減らして品質コストを削減できる」こと
- 原則4はパレートの法則と結びつけ、欠陥の偏在がリスクベースドテストのインプットになる点まで押さえる
- 原則5の対処はテストとテストデータの変更や新規テストの作成だが、自動化されたリグレッションテストのように繰り返しが有益な例外もある
- 原則6から、唯一普遍的に適用できるテストのアプローチは存在しない（状況を問わず通用するアプローチがあるとする選択肢を退ける）
- 原則7は検証だけでは不十分という主張であり、妥当性確認の必要性と対応づけて覚える
