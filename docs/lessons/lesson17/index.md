# lesson17: デシジョンテーブルテスト — 条件の組み合わせを表で網羅する

## このレッスンで学ぶこと

- 複数の条件の組み合わせを扱うデシジョンテーブルテストの目的を理解する
- 条件・アクション・ルールで構成されるデシジョンテーブルの構造と表記を読み書きできるようになる
- 完全なデシジョンテーブルと簡略化・最小化の関係を理解する
- 業務ルールからデシジョンテーブルを作成し、テストケースを導出できるようになる
- デシジョンテーブルテストのカバレッジを計算できるようになる

## デシジョンテーブルテストの考え方

デシジョンテーブルテストは、ブラックボックステスト技法の1つです（技法の分類は [lesson14](/lessons/lesson14/) を参照）。

システム要件には、条件の組み合わせしだいで結果が変わる仕様が数多く登場します。

- 会員かどうかと購入金額の組み合わせで、送料が変わる
- 年齢と免許の有無の組み合わせで、申し込めるプランが変わる
- 入力値の妥当性と権限の組み合わせで、処理の成否が変わる

こうした仕様を文章のまま追いかけると、組み合わせの見落としや解釈の食い違いが起きやすくなります。デシジョンテーブルは、条件のさまざまな組み合わせと、その結果として起きるアクションを1枚の表に整理したものです。ビジネスルールのような複雑なロジックを記録する効果的な方法であり、そうした要件の実装をテストするために使用します。

同値分割法（[lesson15](/lessons/lesson15/)）や境界値分析（[lesson16](/lessons/lesson16/)）が個々の入力に対する値の選び方に役立つのに対し、デシジョンテーブルテストは条件どうしの組み合わせを扱うのに適しています。

一方、条件の組み合わせではなく「状態とイベントの流れ」で振る舞いが決まるシステムには、状態遷移テスト（[lesson18](/lessons/lesson18/)）が適しています。

## デシジョンテーブルの構造

デシジョンテーブルは次の手順で作成します。

1. システムの条件と、結果として起きるアクションを決める
2. 条件とアクションを表の行にする（上部に条件、下部にアクション）
3. 条件の一意な組み合わせと、それに対応するアクションを1つの列にまとめる

この列を**ルール**（rule）と呼びます。ルールは「この条件の組み合わせのとき、このアクションが起きる」という判定を1列で表したものです。

### 条件とアクションの表記

条件の行には、次の値を書きます。

| 表記 | 意味 |
|------|------|
| T | 条件が満たされている（真） |
| F | 条件が満たされていない（偽） |
| —（ダッシュ） | その条件の値がアクションの結果に無関係 |
| N/A | その条件がこのルールでは実行不可能 |

アクションの行には、次の値を書きます。

| 表記 | 意味 |
|------|------|
| X | そのアクションが発生すべき |
| 空白 | そのアクションが発生すべきでない |

これ以外の表記が使われることもありますが、まずはこの基本形を読めるようにしておきましょう。

::: tip T/F 表記で統一する
シラバスの表記は T（真）と F（偽）です。参考書によっては Y/N 表記を使うものもありますが、本教材ではシラバスに合わせて T/F で統一します。
:::

### 制限指定と拡張指定

条件やアクションが取れる値によって、デシジョンテーブルは2種類に分かれます。

| 種類 | 条件・アクションの値 |
|------|--------------------|
| 制限指定デシジョンテーブル | すべてブール値（真または偽）で示す |
| 拡張指定デシジョンテーブル | 一部または全部が複数の値を取る（範囲のある数値、同値パーティション、離散値など） |

拡張指定に出てくる同値パーティションは、同値分割法（[lesson15](/lessons/lesson15/)）の考え方です。このレッスンのワークでは、基本形である制限指定デシジョンテーブルを扱います。

## 完全なデシジョンテーブルと圧縮

完全なデシジョンテーブルは、すべての条件の組み合わせをカバーするのに十分な列を持ちます。真偽で表せる条件がn個なら、列の数は最大で2のn乗です（条件3つなら8列）。

条件が増えると列の数は指数関数的に増えるため、表を圧縮する方法が2つ用意されています。

- 簡略化: 実行不可能な条件の組み合わせを含む列を削除する
- 最小化: いくつかの条件が結果に影響しない列どうしを、「—」を使って1つの列に統合する

最小化のアルゴリズム自体はシラバスの範囲外です。試験では「圧縮した表を読める・作れる」ことが問われると考えて準備しましょう。

## ワーク：送料判定ルールのデシジョンテーブル

具体的な業務ルールを1つ取り上げて、デシジョンテーブルの作成からテストケースの導出までを通して練習します。

::: info 題材とする送料の業務ルール
オンラインショップの送料を次のルールで決める。

- 送料無料クーポンを使った注文は、常に送料無料とする
- クーポンを使わない場合、会員かつ購入金額が5000円以上なら送料無料とする
- それ以外の注文は送料500円とする
:::

### ステップ1：条件とアクションの決定

まず、ルールの文章から条件とアクションを抜き出します。

条件は3つです。

- 条件1: 会員である
- 条件2: 購入金額が5000円以上である
- 条件3: 送料無料クーポンを使用している

アクションは2つです。

- アクション1: 送料を無料にする
- アクション2: 送料500円を請求する

条件は「TかFかで答えられる形」に言い換えるのがコツです。「購入金額」をそのまま条件にするのではなく、「購入金額が5000円以上である」という真偽で判定できる文にします。

### ステップ2：完全なデシジョンテーブルの作成

条件が3つなので、組み合わせは全部で8通り（2の3乗）です。すべての組み合わせを列として書き出します。

| | ルール1 | ルール2 | ルール3 | ルール4 | ルール5 | ルール6 | ルール7 | ルール8 |
|---|---|---|---|---|---|---|---|---|
| 条件1: 会員である | T | T | T | T | F | F | F | F |
| 条件2: 購入金額が5000円以上 | T | T | F | F | T | T | F | F |
| 条件3: クーポンを使用 | T | F | T | F | T | F | T | F |
| アクション1: 送料無料 | X | X | X | | X | | X | |
| アクション2: 送料500円 | | | | X | | X | | X |

列の読み方を2つ確認します。

- ルール2は「会員で、5000円以上で、クーポンなし」の組み合わせです。会員かつ5000円以上なので、送料無料（アクション1にX）です
- ルール6は「非会員で、5000円以上で、クーポンなし」の組み合わせです。会員ではないため無料の条件を満たさず、送料500円（アクション2にX）です

このように全組み合わせを書き出すと、「非会員が5000円以上買った場合はどうなるか」のような、文章では見落としがちなケースにも必ず答えを与えることになります。仕様に答えが書かれていなければ、その時点で要件の漏れを発見できます。

### ステップ3：表の最小化

8列の表をよく見ると、統合できる列があります。

- 条件3がTの列（ルール1・3・5・7）は、条件1と条件2の値に関係なく送料無料です
- 条件3がFで条件1がFの列（ルール6・8）は、条件2の値に関係なく送料500円です

結果に影響しない条件を「—」で表し、これらの列を統合すると、次の4列に圧縮できます。

| | ルール1 | ルール2 | ルール3 | ルール4 |
|---|---|---|---|---|
| 条件1: 会員である | — | T | T | F |
| 条件2: 購入金額が5000円以上 | — | T | F | — |
| 条件3: クーポンを使用 | T | F | F | F |
| アクション1: 送料無料 | X | X | | |
| アクション2: 送料500円 | | | X | X |

最小化後のルール1は元の表のルール1・3・5・7を、最小化後のルール4は元のルール6・8を統合したものです。列の数が8から4に減り、確認すべきルールを絞り込めました。

### ステップ4：テストケースの導出

最低限、実行可能な各ルール（列）を1つのテストケースでカバーします。最小化した表の4つのルールから、テストケースを1つずつ導出します。「—」の条件は結果に影響しないため、現実的な値を1つ選べば十分です。

| テストケース | 会員 | 購入金額 | クーポン | 期待結果 | カバーするルール |
|---|---|---|---|---|---|
| TC1 | 非会員 | 3000円 | 使用する | 送料無料 | ルール1 |
| TC2 | 会員 | 6000円 | 使用しない | 送料無料 | ルール2 |
| TC3 | 会員 | 3000円 | 使用しない | 送料500円 | ルール3 |
| TC4 | 非会員 | 6000円 | 使用しない | 送料500円 | ルール4 |

期待結果が表のアクション行と一致しているかを、1列ずつ突き合わせて確認します。ここまでが「デシジョンテーブルを適用してテストケースを導出する」一連の流れです。

## デシジョンテーブルテストのカバレッジ

デシジョンテーブルテストのカバレッジアイテムは、実行可能な条件の組み合わせを含む列（ルール）です。

カバレッジは「テストケースで通した列の数 ÷ 実行可能な列の総数 × 100」で計算し、パーセンテージで表します。100%にするには、実行可能なすべての列を少なくとも1つのテストケースで通す必要があります。

ワークの例で計算してみましょう。

- 完全な表（8列、すべて実行可能）に対して6列だけテストした場合、カバレッジは6 ÷ 8 × 100で75%です
- 最小化した表（4列）に対してTC1〜TC4を実行した場合、カバレッジは4 ÷ 4 × 100で100%です

## 強みと注意点

デシジョンテーブルテストの強みは次の2つです。

- 見逃される可能性がある組み合わせを含めて、条件の組み合わせのすべてを識別する体系的なアプローチを提供する
- 表に整理する過程で、要件の相違点や矛盾を発見するのに役立つ

一方で、注意点もあります。

- 条件が増えるとルールの数は指数関数的に増えるため、すべてのルールの確認には時間がかかることがある
- ルールが多すぎる場合は、最小化したデシジョンテーブルやリスクベースドアプローチ（[lesson25](/lessons/lesson25/)）で確認するルールの数を減らす

## キーワード

| 用語 | 説明 |
|------|------|
| デシジョンテーブルテスト（decision table testing） | 条件の組み合わせと結果のアクションを表に整理し、各ルールをテストケースでカバーするブラックボックステスト技法 |
| ルール（rule） | デシジョンテーブルの各列。条件の一意な組み合わせと、対応するアクションを定義する |
| 制限指定デシジョンテーブル | 条件とアクションの値をすべてブール値（真または偽）で示す表 |
| 拡張指定デシジョンテーブル | 条件やアクションの一部または全部が複数の値（範囲のある数値、同値パーティション、離散値）を取る表 |
| 完全なデシジョンテーブル | すべての条件の組み合わせをカバーするのに十分な列を持つ表 |
| 簡略化 | 実行不可能な条件の組み合わせを含む列を削除して、表を圧縮すること |
| 最小化 | 結果に影響しない条件を持つ列どうしを1つの列に統合して、表を圧縮すること |

## 試験のポイント

- 4.2.3はK3のため、仕様からデシジョンテーブルを作成し、表からテストケースを導出し、カバレッジを計算する適用問題に備える
- 表の構造は「条件とアクションが行、各列がルール」で、行と列を入れ替えた説明はひっかけ
- 表記の読み書きが基本（条件はT・F・—・N/A、アクションはXと空白で、シラバスの表記はY/NではなくT/F）
- 真偽で表せる条件がn個なら、完全なデシジョンテーブルの列は最大で2のn乗になる（条件3つなら8列という列数の計算が頻出）
- 圧縮の2つの方法を区別する（簡略化は実行不可能な組み合わせを含む列の削除、最小化は「—」を使った列の統合で、最小化のアルゴリズム自体はシラバスの範囲外）
- カバレッジ計算の勘所は分母が「実行可能な」列の総数であることで、100%には実行可能な各ルールを最低1つのテストケースでカバーする
- 強み（組み合わせの体系的な識別・要件の相違点や矛盾の発見）と注意点（条件の増加によるルール数の指数関数的な増加）を対で押さえる
- 条件の組み合わせで結果が決まる仕様にはデシジョンテーブルテスト、状態とイベントの流れで振る舞いが決まるシステムには状態遷移テスト（[lesson18](/lessons/lesson18/)）という使い分け
