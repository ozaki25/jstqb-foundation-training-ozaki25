# 用語集

JSTQB Foundation Level で押さえておきたい重要用語を、全 30 レッスンのキーワードから章別に整理しました。用語の表記・定義はシラバス（V4.0）に準拠しています。

::: tip 使い方
各用語の説明末には、初出レッスンへのリンクを付けています。より詳しい説明や具体例はリンク先のレッスンで確認してください。より正確な定義は公式シラバスおよび ISTQB 用語集を参照してください。
:::

## 第1章 テストの基礎

### テストの基本概念

| 用語 | 説明 |
|------|------|
| テスト（testing） | 欠陥を発見し、テスト対象の品質を評価するための一連の活動。テスト実行以外の活動も含む（[lesson01](/lessons/lesson01/)） |
| テスト対象（test object） | テストの対象となるソフトウェアアーティファクト（要件、設計、コード、システムなど）（[lesson01](/lessons/lesson01/)） |
| 検証（verification） | 指定されている要件をシステムが満たしているかを確認すること（[lesson01](/lessons/lesson01/)） |
| 妥当性確認（validation） | ユーザーやその他のステークホルダーのニーズを運用環境でシステムが満たしているかを確認すること（[lesson01](/lessons/lesson01/)） |
| デバッグ（debugging） | 故障の原因である欠陥を発見し、解析し、取り除く活動。テストとは別の活動（[lesson01](/lessons/lesson01/)） |
| カバレッジ（coverage） | テスト対象をテストでどの程度網羅できたかを示す度合い（[lesson01](/lessons/lesson01/)） |

### エラー・欠陥・故障

| 用語 | 説明 |
|------|------|
| エラー（誤り、error） | 人間の誤った行為や判断。欠陥を生み出すことがある（[lesson02](/lessons/lesson02/)） |
| 欠陥（defect、バグ） | 作業成果物に含まれる不備。実行されると故障を引き起こすことがある（[lesson02](/lessons/lesson02/)） |
| 故障（failure） | コンポーネントやシステムが期待通りに動作しない事象。環境条件によって発生することもある（[lesson02](/lessons/lesson02/)） |
| 根本原因（root cause） | 問題が発生する根底の理由。取り除くことで同様の故障や欠陥の防止や頻度低減につながる（[lesson02](/lessons/lesson02/)） |
| 根本原因分析（root cause analysis） | 根本原因を特定する分析。故障の発生時や欠陥の確認時に行うのが一般的（[lesson02](/lessons/lesson02/)） |
| 品質コントロール（QC、quality control） | 適切な品質の達成を支援する活動に焦点を当てた、プロダクト指向の是正アプローチ。テストは QC の主要な形式（[lesson02](/lessons/lesson02/)） |
| 品質保証（QA、quality assurance） | プロセスの実装と改善に焦点を当てた、プロセス指向の予防的アプローチ。プロジェクトに参加する全員が責任を持つ（[lesson02](/lessons/lesson02/)） |

### テストの7原則

| 用語 | 説明 |
|------|------|
| テストの7原則 | あらゆるテストに適用できる一般的なガイドラインとしてシラバスが示す7つの原則（[lesson03](/lessons/lesson03/)） |
| 全数テスト | すべての入力や条件の組み合わせをテストすること。ごく単純なソフトウェア以外では非現実的（[lesson03](/lessons/lesson03/)） |
| 早期テスト | 静的テストと動的テストをなるべく早い時期に開始すること。品質コストの削減につながる（[lesson03](/lessons/lesson03/)） |
| 欠陥の偏在 | 欠陥や運用時の故障の大部分が少数のシステムコンポーネントに集中する現象（[lesson03](/lessons/lesson03/)） |
| パレートの法則 | 結果の大部分が少数の原因から生じるという経験則。欠陥の偏在はその一例（[lesson03](/lessons/lesson03/)） |
| テストの弱化 | 同じテストの繰り返しで新たな欠陥の検出効果が薄れること。旧称は「殺虫剤のパラドックス」（[lesson03](/lessons/lesson03/)） |
| 「欠陥ゼロ」の落とし穴 | 検出した欠陥をすべて修正しても、ニーズを満たす有用なシステムになるとは限らないこと（[lesson03](/lessons/lesson03/)） |

### テストプロセス

| 用語 | 説明 |
|------|------|
| テストプロセス（test process） | テスト目的を達成する確率を高める、汎用的なテスト活動のセット。コンテキストに合わせて調整できる（[lesson04](/lessons/lesson04/)） |
| テスト計画（test planning） | テスト目的を定義し、制約の中で目的を最も効果的に達成するアプローチを選択する活動（[lesson04](/lessons/lesson04/)） |
| テストモニタリング（test monitoring） | すべてのテスト活動を継続的にチェックし、実際の進捗をテスト計画と比較する活動（[lesson04](/lessons/lesson04/)） |
| テストコントロール（test control） | テストの目的を達成するために必要な行動をとる活動（[lesson04](/lessons/lesson04/)） |
| テスト分析（test analysis） | テストベースを分析してテスト可能なフィーチャーを識別し、テスト条件を定義して優先順位を付ける活動。「何をテストするか」に答える（[lesson04](/lessons/lesson04/)） |
| テスト設計（test design） | テスト条件をテストケースなどのテストウェアに落とし込む活動。「どのようにテストするか」に答える（[lesson04](/lessons/lesson04/)） |
| テスト実装（test implementation） | テスト実行に必要なテストウェアを作成または取得し、テスト環境を構築する活動（[lesson04](/lessons/lesson04/)） |
| テスト実行（test execution） | テスト実行スケジュールに従ってテストを走らせ、実際の結果と期待結果を比較する活動（[lesson04](/lessons/lesson04/)） |
| テスト完了（test completion） | プロジェクトのマイルストーンでテストウェアの引き渡しや教訓の識別を行い、テスト完了レポートを伝える活動（[lesson04](/lessons/lesson04/)） |
| テストベース（test basis） | テスト条件やテストケースを導出するもとになる情報（要件、設計、コードなど）（[lesson04](/lessons/lesson04/)） |
| テスト条件（test condition） | テスト分析で識別する、テストで確認すべきテスト対象の側面（[lesson04](/lessons/lesson04/)） |
| テストケース（test case） | テスト条件に基づいて作成する、入力・事前条件・期待結果などのセット（[lesson04](/lessons/lesson04/)） |
| テストプロシジャー（test procedure） | テストケースを実行の観点で編成したもの。多くの場合テストスイートにまとめる（[lesson04](/lessons/lesson04/)） |
| テストウェア（testware） | テスト活動の出力となる作業成果物（テスト計画書、テストケース、テストデータ、テスト結果記録、テストレポートなど）（[lesson04](/lessons/lesson04/)） |
| トレーサビリティ（traceability） | テストベースの各要素とテストウェア・テスト結果・検出した欠陥との間のつながりを追跡できること（[lesson04](/lessons/lesson04/)） |

### テストに必要なスキルとチーム

| 用語 | 説明 |
|------|------|
| スキル（skill） | 知識・実践・適性に基づいて何かをうまく行う能力（[lesson05](/lessons/lesson05/)） |
| 確証バイアス（confirmation bias） | 現在持っている信念と異なる情報を受け入れにくくなる傾向。欠陥報告の受け止め方に影響する（[lesson05](/lessons/lesson05/)） |
| チーム全体アプローチ（whole-team approach） | 必要な知識とスキルを持つメンバーなら誰がどのタスクを担当してもよく、全員が品質に責任を持つという考え方。XP に由来する（[lesson05](/lessons/lesson05/)） |
| エクストリームプログラミング（XP） | アジャイル開発手法の1つ。チーム全体アプローチの由来（[lesson05](/lessons/lesson05/)） |
| テストの独立性（independence of testing） | テストを行う人が作業成果物の作成者からどれだけ離れているかの度合い。認知バイアスの違いにより欠陥発見の効果を高める（[lesson05](/lessons/lesson05/)） |

## 第2章 ソフトウェア開発ライフサイクル全体を通してのテスト

### 開発モデルとテストファーストアプローチ

| 用語 | 説明 |
|------|------|
| SDLCモデル（software development lifecycle model） | ソフトウェア開発プロセスを抽象的かつハイレベルに表現したもの。開発フェーズや活動の種類の論理的・時系列的な関連を定義する（[lesson06](/lessons/lesson06/)） |
| シーケンシャル開発モデル（sequential development model） | 開発フェーズを順に進める SDLC モデル。ウォーターフォールモデルや V 字モデルが例（[lesson06](/lessons/lesson06/)） |
| イテレーティブ開発モデル（iterative development model） | 開発を繰り返しのサイクルで進める SDLC モデル。スパイラルモデルやプロトタイピングが例（[lesson06](/lessons/lesson06/)） |
| インクリメンタル開発モデル（incremental development model） | システムを増分として少しずつ積み上げる SDLC モデル。統一プロセスが例（[lesson06](/lessons/lesson06/)） |
| テストファーストアプローチ（test-first approach） | コードを書く前にテストケースを定義するアプローチ。TDD・ATDD・BDD が該当する（[lesson06](/lessons/lesson06/)） |
| TDD（テスト駆動開発） | テストケースを通じてコーディングを導くアプローチ。テストを書き、テストを満たすコードを書き、リファクタリングする（[lesson06](/lessons/lesson06/)） |
| BDD（振る舞い駆動開発） | 望ましい振る舞いを自然言語のテストケースで表現するアプローチ。通常は Given/When/Then 形式を使う（[lesson06](/lessons/lesson06/)） |

### DevOpsとシフトレフト

| 用語 | 説明 |
|------|------|
| DevOps | 開発（テストを含む）と運用が共通のゴールの達成に向けて協力し、相乗効果を生み出すことを目指す組織的アプローチ（[lesson07](/lessons/lesson07/)） |
| 継続的インテグレーション（CI） | コードのサブミットのたびに、ビルド・静的解析・自動コンポーネントテストなどを自動で実行する技術的な実践（[lesson07](/lessons/lesson07/)） |
| 継続的デリバリー（CD） | ビルドからリリースまでの流れを自動化し、ソフトウェアをいつでもリリースできる状態に保つ技術的な実践（[lesson07](/lessons/lesson07/)） |
| デリバリーパイプライン | ビルド・テスト・リリースの流れを自動化した一連の仕組み。DevOps で高品質なコードを速く届ける土台となる（[lesson07](/lessons/lesson07/)） |
| シフトレフト（shift left） | 早期テストの原則にもとづき、テストを SDLC のより早い段階で行うアプローチ（[lesson07](/lessons/lesson07/)） |
| ふりかえり（retrospective） | 何が成功したか・何を改善できるかをチームで話し合い、プロセスの継続的改善につなげるミーティング（[lesson07](/lessons/lesson07/)） |

### テストレベル

| 用語 | 説明 |
|------|------|
| テストレベル（test level） | 系統的にまとめてマネジメントしていくテスト活動のグループ。開発段階に応じたテストプロセスのインスタンス（[lesson08](/lessons/lesson08/)） |
| コンポーネントテスト（component testing） | コンポーネントを単独でテストすることに焦点を当てるテストレベル。ユニットテストとも呼ぶ（[lesson08](/lessons/lesson08/)） |
| コンポーネント統合テスト（component integration testing） | コンポーネント間のインターフェースと相互処理に焦点を当てるテストレベル。ユニット統合テストとも呼ぶ（[lesson08](/lessons/lesson08/)） |
| システムテスト（system testing） | システムやプロダクト全体の振る舞いや能力の全般に焦点を当てるテストレベル（[lesson08](/lessons/lesson08/)） |
| システム統合テスト（system integration testing） | テスト対象システムと他のシステムや外部サービスとのインターフェースに焦点を当てるテストレベル（[lesson08](/lessons/lesson08/)） |
| 受け入れテスト（acceptance testing） | 妥当性確認と、デプロイの準備ができていることの実証に焦点を当てるテストレベル（[lesson08](/lessons/lesson08/)） |
| ユーザー受け入れテスト（user acceptance testing、UAT） | 想定ユーザーがビジネスニーズの充足を確認する受け入れテストの形態（[lesson08](/lessons/lesson08/)） |
| 運用受け入れテスト（operational acceptance testing） | 運用担当者やシステム管理者が運用面を確認する受け入れテストの形態（[lesson08](/lessons/lesson08/)） |
| アルファテスト（alpha testing） | 開発組織側の環境で実施する受け入れテストの形態（[lesson08](/lessons/lesson08/)） |
| ベータテスト（beta testing） | ユーザー自身の環境で実施する受け入れテストの形態（[lesson08](/lessons/lesson08/)） |
| テストハーネス（test harness） | コンポーネントを単独で実行するために、スタブやドライバーなどを組み合わせて用意するテスト環境（[lesson08](/lessons/lesson08/)） |

### テストタイプと確認・リグレッションテスト

| 用語 | 説明 |
|------|------|
| テストタイプ（test type） | 特定の観点や品質特性に着目したテスト活動のまとまり。すべてのテストレベルに適用できる（[lesson09](/lessons/lesson09/)） |
| 機能テスト（functional testing） | テスト対象が「何」をすべきかを評価するテスト。機能完全性・機能正確性・機能適切性をチェックする（[lesson09](/lessons/lesson09/)） |
| 非機能テスト（non-functional testing） | システムが「どのようにうまく振る舞うか」を評価するテスト。性能効率性・使用性・信頼性・セキュリティなどの品質特性をチェックする（[lesson09](/lessons/lesson09/)） |
| ISO/IEC 25010 | 非機能ソフトウェア品質特性を分類している標準。性能効率性・互換性・使用性・信頼性・セキュリティ・保守性・移植性を挙げている（[lesson09](/lessons/lesson09/)） |
| ブラックボックステスト（black-box testing） | 仕様に基づき、テスト対象の外から見える振る舞いを示すドキュメントからテストを導出するテスト（[lesson09](/lessons/lesson09/)） |
| ホワイトボックステスト（white-box testing） | 構造に基づき、コードやアーキテクチャーなどの実装・内部構造からテストを導出するテスト（[lesson09](/lessons/lesson09/)） |
| 確認テスト（confirmation testing） | 元の欠陥が正常に修正されたことを確認するテスト（[lesson09](/lessons/lesson09/)） |
| リグレッションテスト（regression testing） | すでに確認テスト済みの修正を含め、変更が悪影響（デグレード）を引き起こしていないことを確認するテスト（[lesson09](/lessons/lesson09/)） |
| 影響度分析（impact analysis） | ソフトウェアのどの部分が変更の影響を受けるかを示す分析。リグレッションテストの範囲の最適化に役立つ（[lesson09](/lessons/lesson09/)） |

### メンテナンステスト

| 用語 | 説明 |
|------|------|
| メンテナンステスト（maintenance testing） | 運用中のシステムやその環境への変更に対して行うテスト。計画的なリリース/デプロイと計画外のリリース/デプロイの両方を含む（[lesson10](/lessons/lesson10/)） |
| メンテナンス（maintenance、保守） | リリース後のシステムに変更を加える活動。修正を伴うもの、環境の変化に適応するもの、パフォーマンスや保守性を改善するものなどがある（[lesson10](/lessons/lesson10/)） |
| ホットフィックス（hotfix） | 計画外のリリース/デプロイによる緊急の修正（[lesson10](/lessons/lesson10/)） |

## 第3章 静的テスト

### 静的テストの基礎

| 用語 | 説明 |
|------|------|
| 静的テスト（static testing） | ソフトウェアを実行せずに作業成果物を評価するテスト。レビューと静的解析を含む（[lesson11](/lessons/lesson11/)） |
| 動的テスト（dynamic testing） | ソフトウェアの実行を伴うテスト。故障を引き起こし、その後の分析によって欠陥を特定する（[lesson11](/lessons/lesson11/)） |
| 静的解析（static analysis） | ツールを用いた作業成果物の評価。テストケースが不要で、対象には構造（モデル、コード、正式な構文を持つテキストなど）が必要（[lesson11](/lessons/lesson11/)） |
| 作業成果物（work product） | 開発やテストの過程で作成される成果物。要件仕様書・ソースコード・テスト計画書・契約書・モデルなど（[lesson11](/lessons/lesson11/)） |

### レビュープロセス

| 用語 | 説明 |
|------|------|
| レビュー（review） | 作業成果物を評価してフィードバックを得る、静的テストの一種（[lesson12](/lessons/lesson12/)） |
| 不正（anomaly） | 期待から逸脱していると識別された事項。分析と議論の結果、欠陥と判断されるとは限らない（[lesson12](/lessons/lesson12/)） |
| マネージャー（manager） | 何をレビューするかを決定し、人員や時間などのリソースを提供する役割（[lesson12](/lessons/lesson12/)） |
| 作成者（author） | レビュー対象の作業成果物を作成し、修正する役割（[lesson12](/lessons/lesson12/)） |
| モデレーター（moderator） | レビューミーティングを効果的に運営する役割。ファシリテーターとも呼ばれる（[lesson12](/lessons/lesson12/)） |
| 書記（scribe） | 不正の取りまとめとレビュー情報の記録を担う役割。記録係とも呼ばれる（[lesson12](/lessons/lesson12/)） |
| レビューア（reviewer） | 作業成果物の品質を精査し、レビューを実施する役割（[lesson12](/lessons/lesson12/)） |
| レビューリーダー（review leader） | 参加者と開催の日時・場所を決め、レビューの全体的な責任を負う役割（[lesson12](/lessons/lesson12/)） |
| チェックリストベースドレビュー（checklist-based reviewing） | あらかじめ用意した項目のリストを手がかりに作業成果物を精査するレビュー技法（[lesson12](/lessons/lesson12/)） |
| シナリオベースドレビュー（scenario-based reviewing） | 想定するシナリオに沿って作業成果物を精査するレビュー技法（[lesson12](/lessons/lesson12/)） |

### レビュー種別

| 用語 | 説明 |
|------|------|
| 非形式的レビュー（informal review） | 定義されたプロセスに従わず、正式にドキュメント化されたアウトプットを必要としないレビュー。主な目的は不正の検出（[lesson13](/lessons/lesson13/)） |
| ウォークスルー（walkthrough） | 作成者が主導するレビュー。教育・合意形成・新しいアイデアの創出など、幅広い目的を達成できる（[lesson13](/lessons/lesson13/)） |
| テクニカルレビュー（technical review） | 技術的に適格なレビューアが実施し、モデレーターが主導するレビュー。技術的な問題に関する合意と判断をめざす（[lesson13](/lessons/lesson13/)） |
| インスペクション（inspection） | 最も形式的なレビュー。汎用のレビュープロセスに完全に従い、不正の最大数の発見を主な目的とする。メトリクスを収集して SDLC の改善に使用する（[lesson13](/lessons/lesson13/)） |
| 形式的レビュー（formal review） | 定義されたプロセスに従い、正式にドキュメント化されたアウトプットを伴うレビュー。インスペクションが代表例（[lesson13](/lessons/lesson13/)） |

## 第4章 テスト分析と設計

### テスト技法の種類

| 用語 | 説明 |
|------|------|
| テスト技法（test technique） | テスト分析とテスト設計を支援する手法。比較的少ないながらも十分なテストケースのセットを体系的に開発するのに役立つ（[lesson14](/lessons/lesson14/)） |
| ブラックボックステスト技法（black-box test technique） | 仕様ベース技法。内部構造を参照せず、仕様に基づく振る舞いの分析からテストを導出する（[lesson14](/lessons/lesson14/)） |
| ホワイトボックステスト技法（white-box test technique） | 構造ベース技法。テスト対象の内部構造や処理の分析からテストを導出する（[lesson14](/lessons/lesson14/)） |

### 同値分割法

| 用語 | 説明 |
|------|------|
| 同値分割法（equivalence partitioning、EP） | すべての要素が同等に処理されることを想定してデータをパーティションに分割し、各パーティションから代表値を選んでテストするブラックボックステスト技法（[lesson15](/lessons/lesson15/)） |
| 同値パーティション（equivalence partition） | 同じように処理されると想定される値の集合。同値クラスとも呼ぶ。重複せず、空でない集合でなければならない（[lesson15](/lessons/lesson15/)） |
| 有効パーティション（valid partition） | 有効な値を包含するパーティション。テスト対象が処理すべき値、仕様書に処理が定義されている値と解釈できる（[lesson15](/lessons/lesson15/)） |
| 無効パーティション（invalid partition） | 無効な値を包含するパーティション。テスト対象が無視または拒絶すべき値、仕様書に処理が定義されていない値と解釈できる（[lesson15](/lessons/lesson15/)） |
| イーチチョイスカバレッジ（each choice coverage） | 複数のパーティションセットがある場合に、各セットの各パーティションを少なくとも1回テストケースで通すことを要件とする基準。組み合わせは考慮しない（[lesson15](/lessons/lesson15/)） |

| リスクモニタリング（risk monitoring） | 軽減措置の効果の確認、アセスメント改善のための情報の入手、新たなリスクの識別を行う活動（[lesson25](/lessons/lesson25/)） |
| リスクベースドテスト（risk-based testing） | リスク分析とリスクコントロールに基づいて、テスト活動を選択し、優先順位を付け、マネジメントするテストアプローチ（[lesson25](/lessons/lesson25/)） |

### テストのモニタリングとコントロール

| 用語 | 説明 |
|------|------|
| テストメトリクス（test metrics） | 進捗状況・テスト対象の品質・テスト活動の有効性を示すために収集する測定値（[lesson26](/lessons/lesson26/)） |
| テスト進捗レポート（test progress report） | テストのモニタリングとコントロールの間に定期的に作成し、テストの継続的なコントロールを支援するレポート（[lesson26](/lessons/lesson26/)） |
| テスト完了レポート（test completion report） | テスト完了時に作成し、テストの特定の段階を要約して、その後のテストのための情報を提供するレポート（[lesson26](/lessons/lesson26/)） |
| 欠陥密度（defect density） | 欠陥メトリクスの例。成果物の規模あたりの欠陥数を示す（[lesson26](/lessons/lesson26/)） |
| 残存リスクレベル（residual risk level） | リスクメトリクスの例。テスト後になお残っているリスクのレベル（[lesson26](/lessons/lesson26/)） |

### 構成管理

| 用語 | 説明 |
|------|------|
| 構成管理（configuration management、CM） | テスト計画書、テストケース、テスト結果記録などの作業成果物を、構成アイテムとして識別、コントロール、トラッキングするための規律（[lesson27](/lessons/lesson27/)） |
| 構成アイテム（configuration item） | 構成管理の対象として扱う作業成果物や構成要素。テストケース、テストスクリプト、テスト環境など（[lesson27](/lessons/lesson27/)） |
| ベースライン（baseline） | テスト用に承認された構成アイテム。正式な変更コントロールプロセスを通じてのみ変更できる（[lesson27](/lessons/lesson27/)） |
| バージョンコントロール（version control） | 構成アイテムのバージョンを管理し、変更を追跡すること（[lesson27](/lessons/lesson27/)） |

### 欠陥マネジメント

| 用語 | 説明 |
|------|------|
| 欠陥マネジメント（defect management） | 報告された不正を記録・解析・分類し、対応を決定してクローズまで追跡する活動（[lesson28](/lessons/lesson28/)） |
| 欠陥レポート（defect report） | 報告された不正（欠陥）の情報をまとめ、解決までの追跡に使うドキュメント。ISO/IEC/IEEE 29119-3 ではインシデントレポートと呼ぶ（[lesson28](/lessons/lesson28/)） |
| 偽陽性（false positive） | 実際には欠陥が存在しないのに、欠陥として報告されたもの（[lesson28](/lessons/lesson28/)） |
| 重要度（severity） | ステークホルダーの利益や要件に対する欠陥の影響の度合い（[lesson28](/lessons/lesson28/)） |
| 優先度（priority） | 欠陥を修正する緊急度（[lesson28](/lessons/lesson28/)） |
| 欠陥のライフサイクル | 欠陥レポートが報告からクローズまでにたどる状態の遷移（[lesson28](/lessons/lesson28/)） |

## 第6章 テストツール

### テストツールとテスト自動化

| 用語 | 説明 |
|------|------|
| テストツール（test tool） | テスト活動を支援、促進するツール。テスト専用ツールに限らず、テストのコンテキストで使う汎用ツールも含む（[lesson29](/lessons/lesson29/)） |
| コンテナ化ツール（containerization tool） | 拡張性と展開の標準化を支援するツールの例。仮想マシンと並んで挙げられる（[lesson29](/lessons/lesson29/)） |
| テスト自動化（test automation） | ツールを使用してテスト活動を実行または支援すること。利点とリスクの両方を分析したうえで導入する（[lesson30](/lessons/lesson30/)） |
| テストスクリプト | 自動テストの実行手順を記述したもの。テスト対象の変更に合わせた保守が必要になる（[lesson30](/lessons/lesson30/)） |
