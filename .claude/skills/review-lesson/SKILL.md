# レッスンレビュースキル

指定されたレッスンを、下記チェックリストの**全カテゴリ**で必ずレビューします。
場当たりに気づいた点だけ直すのは禁止。**毎回チェックリストを頭から最後まで通す**こと。

## 基準は CLAUDE.md とシラバス

このチェックリストは運用の補助です。**品質の基準は `CLAUDE.md` の執筆方針全体、および JSTQB/ISTQB シラバスそのもの**です。

- レビュー開始時に、**毎回 `CLAUDE.md`（存在する場合）の執筆方針を最初から最後まで読む**こと。記憶や下のチェックリストだけに頼らない
- 下のチェックリストに**載っていない方針が CLAUDE.md にあれば、それも必ず適用する**（チェックリストの記載漏れより CLAUDE.md が優先）
- CLAUDE.md の方針が更新されたら、**このスキルのチェックリストにも反映する**

## 引数

- `$ARGUMENTS` にレビュー対象（`lesson07` など）
- 複数指定された場合は1本ずつ順にループを回す

## レビューのループ（仕組み）

1 本のレッスン（`docs/lessons/lessonNN/index.md`）について、次を**指摘ゼロになるまで繰り返す**:

```
① 対象ファイルを最初から最後まで通読する
② 下のチェックリストを A→I の順に全項目チェックし、指摘を箇条書きで列挙する
③ 指摘を修正する
④ npm run docs:build でビルドが通ることを確認する
⑤ ②に戻る。指摘が出なくなったら、その時点で1周「指摘なし」を確認して終了
```

**1周でもチェックを飛ばさない。** 修正で別の問題を作ることがあるため、修正後は必ず全項目を再チェックする。

最後に、各カテゴリについて「指摘なし / 指摘あり（内容と対応）」を表で報告する。

## チェックリスト

### A. 仕組みの正確性・最新版（最優先）

- **JSTQB/ISTQB シラバスを一次基準に照合したか。これはこのプロジェクトの品質の核心である。** 用語・定義・分類・図の構造をシラバスの表記に厳密準拠させる（例: デシジョンテーブルは「条件」「アクション」を行に、各列を「ルール」とする。状態遷移テストは「状態」「イベント」「遷移」「アクション」で説明する。リスクは「リスクの可能性」×「リスクの影響」と表記する。同値分割・境界値分析などの技法名・手順名もシラバスの訳語をそのまま使う）
- **二次情報（解説記事・Web 検索の要約）は入口にとどめ、用語の表記までシラバス本文（一次情報）で確認する**。二次情報だけに頼って表記が条文と食い違う事故を避ける（例: 「発生可能性/影響度」ではなく「リスクの可能性/リスクの影響」、独自の意訳語ではなくシラバス訳語を使う）
- 章・節の対応関係（このレッスンがシラバスのどの節に対応するか）が明確か。範囲外の内容を混ぜて範囲を逸脱していないか
- 用語が複数の言い方を持つ場合、シラバスで採用されている訳語を優先しているか
- 同じ用語・技法を扱う他レッスンと矛盾していないか（例: 同値分割の説明が lesson15 と lesson17 で食い違っていないか）

### B. 構成・設計

- **単一の核**になっているか。複数の核が混在して1本に詰め込まれていないか
- **仕組み・背景から語っているか**。表面的な紹介（「〜というものがある」）で終わらず、なぜそれが必要か・どう使うかから説明しているか
- ピラミッド構造（テーマ → 柱3つ程度 → 各柱の詳細）で上位から整理されているか
- **レッスン単体で完結**しているか。「前回」「lessonXX で」など他レッスンへの参照がないか。前提知識はその場で簡潔に説明しているか
- **15分で読み切れるボリューム**か。網羅性より「今日はこれだけ覚えれば OK」を優先しているか
- **分類・一覧系のレッスンは、挙げた項目すべてに実体があるか**（例: 「技法は A・B・C」と挙げたらそれぞれに本文が対応しているか）。挙げたのに中身が無い、という不整合を作らない

### C. レッスンの型（学ぶこと・キーワード・試験のポイント）

- `## このレッスンで学ぶこと` が **3〜5項目**あるか。各項目が**1行・1文以内**の短いフレーズで、本文のような説明を書き込んでいないか
  - **句点「。」の数を機械的に数える**。1 項目に「。」が 2 つ以上、または「。」で区切られた 2 文が入っていたら不可（例:「全部はテストできない。だから優先順位をつける」→「全部はテストできないから優先順位をつける」と 1 文に繋ぐ）
- `## キーワード` が用語と説明の表になっているか。シラバスの用語がそのまま（表記ゆれなく）使われているか。説明が簡潔（1行程度）か
- `## 試験のポイント` が箇条書きで、出題されやすい観点（定義の言い換え、混同しやすい用語の区別、数値・順序など）を押さえているか。各項目が複文になっていないか（C の句点チェックと同様に確認する）
- 本文の説明の丸写しになっていないか（キーワード・試験のポイントは想起のトリガーに徹する）
- 項目が多いとき、分割ではなく統合で減らしているか

### D. 文章品質（直訳調・AI 詩的文体・自然な日本語）

- **`natural-japanese` スキルを必ず適用する**。日本語の「AIっぽさ」（不自然な言い回し、機械的な直訳調、AI が書いたと分かる特徴的なパターン）が無いか、このスキルの観点で通読すること
- 直訳調: 名詞化の硬さ（「〜の多さ」「〜の欠如」）、不自然な受動態、回りくどい修飾がないか
- 英語由来のラベル語（「落とし穴」「柱」「層」）を見出し・番号ラベルに裸で使っていないか
- **カタカナ専門用語の安易な採用**がないか。シラバスに定訳がある場合はそれを使い、前文で平易に言い換えた直後に急にカタカナ・漢語へ切り替えていないか
- AI 詩的文体: 体言止めの列挙、短文の連打でリズムを作る、問いかけ+即答、ドラマチックな一文で締める、がないか
- **「説明の足場」が残っていないか**（CLAUDE.md 参照）。メタな前置き／数え上げの宣言（「違いは大きく N つです」→「1 つ目は」）／番号ラベル見出し（「違い 1:」）／教科書的な締め（「〜という設計です」）／不自然な比喩（「〜の地図」「正体」）。**各段落を「同僚に口頭でこう言うか」で判定する**。言わない言い回しは足場なので削る
- ダッシュ（——）を**文中**に使っていないか（h1 の — 区切りは可）
- **括弧書き・記号（＝・＋）を多用していないか**。装飾的な「（…）」は文に溶かす。括弧は「後で使う用語・シラバス上の対訳語の補足」「短い例の列挙」に限る。「A＝B」「A ＋ B」は言葉（「A は B」「A に B を足す」）で書く
- 3文以上続く段落・複数論点の段落を、短文・箇条書き・表・図に分解しているか

### E. 見出し

- `# lessonNN: テーマ — サブタイトル` の形式（NN はゼロ埋め2桁）になっているか
- `##` 以降の見出しは名詞句で簡潔か。語りかけ調（「〜しよう」「〜ですよね」「〜してみる」）になっていないか
- 詩的・文学的な語（正体／裏側／命を吹き込む／鍵／秘密／魔法／旅／世界／扉／真実）を使っていないか。中身を表す具体語にする

### F. ペルソナ

- 読者（**テスト実務の経験が浅い、これから JSTQB Foundation Level 受験を目指す社会人・エンジニア**）が共感できるか
- 実務経験や高度な専門知識を前提にした書き方をしていないか。初めて聞く用語として丁寧に導入しているか
- 専門用語を初出時に平易な言葉で説明しているか
- 「試験でどう問われるか」を意識した説明になっているか（暗記ではなく理解して答えられる書き方）

### G. 例・図の質

- 具体例（デシジョンテーブルの表、状態遷移図、テストケース例など）が断片でなく、文脈のわかる完結した例か
- 表・箇条書きが実務やシラバスの慣習に沿った項目立てになっているか

### H. 図・デモ

- 図は Mermaid（` ```mermaid `）で書かれているか。ASCII アートを使っていないか
- **図はコードを読むだけで判定せず、実際に画像へレンダリングして目視する**。ライトとグレースケールの両方で、重なり・はみ出し・ラベルの読みやすさ・色依存を目で確認する
- **図が色だけで区別していないか**。意味の区別は色に依存させず、必ず形・記号・パターン・位置・ラベルなど色以外の手がかりを併用する。グレースケールにしても読めるかで判定する
- 表が横スクロールで破綻していないか

### I. ビルド・機械チェック

- `npm run docs:build` が通るか
- `npm run docs:lint`（textlint）が通るか
- `npm run quiz:validate` が関連する場合は通るか
- **`node .claude/hooks/check-style.js docs/lessons/lessonNN/index.md` で型の欠陥（試験のポイント・学ぶことの複文、詩的見出し、全角記号 ＝＋、足場フレーズ）を機械検出したか**。候補が出たら引用・固有名詞・シラバス上の正当な表記かを人が判断して、該当すれば直す（無引数で `docs/lessons` 配下の全 index.md を一括スキャンできる）
- **`node .claude/hooks/check-lesson-title.js docs/lessons/lessonNN/index.md` で h1 がファイル名の lessonNN と一致した形式か確認したか**
- **太字 `**` が素通りしていないか**。markdown-it は全角約物（（）「」など）に隣接した `**` を、文中（前後が日本語）にあるとき太字にしないため、目視では気づけない。`node .claude/hooks/check-bold.js docs/lessons/lessonNN/index.md` で検査し、出力が空であることを確認する。崩れたら約物を `**` の外に出す（例: `**目次（一覧）**` → `**目次**（一覧）`）

## 出力フォーマット

各レッスンについて、最終的に次を報告する:

```
## lessonNN レビュー結果

| カテゴリ | 判定 | 指摘・対応 |
|---------|------|-----------|
| A シラバス整合 | ✅/⚠️ | ... |
| B 構成 | ... | ... |
| ...（C〜I まで全カテゴリ必ず記載） |

修正したコミット: <ハッシュ>
```

カテゴリを省略しない。指摘がなくても「✅ 指摘なし」と明記する。
