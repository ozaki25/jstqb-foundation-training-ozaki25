# CLAUDE.md

このリポジトリで作業する際の方針をまとめます。

## プロジェクト概要

**JSTQB Foundation Level**（テスト技術者資格 Foundation Level）の合格を目指す、**非公式・非公認**の学習コンテンツです。全 6 章 30 レッスンで構成し、1 レッスンがシラバスの小カテゴリ（またはテスト技法）に対応するよう設計しています。開発の経緯は `SPRINTLOG.md` を参照。

現状: 全 30 レッスンの本文とドリル 186 問（1 レッスン 5〜7 問）が完成済み。以降は `FACTCHECK.md` の裏取りと横断的な品質改善を継続する。

- サイトは [VitePress](https://vitepress.dev/) で構築し、Vercel にデプロイします。
- 参考にした既存リポジトリ: `ux-certification-basics-ozaki25`（同じ仕組みの検定学習コンテンツ）。

## 試験の前提

- 実施: 一般社団法人 日本語ソフトウェアテスト資格認定委員会（JSTQB）。公式: https://www.jstqb.jp/syllabus/
- 一次資料: **JSTQB Foundation Level シラバス Version 2023 V4.0.J02**（全 6 章）。これが出題範囲の正。
- 形式: 選択式・**40 問・60 分・合格ライン 65%（40 問中 26 問）**。
- 各学習目標には認知レベル **K1（記憶）・K2（理解）・K3（適用）** が設定されている。K3 の技法（同値分割法・境界値分析・デシジョンテーブルテスト・状態遷移テスト・ATDD・見積り・優先順位付け・欠陥レポート作成）は「使える」ことが問われるため、手順と適用例を必ず載せる。
- コンテンツを書く前に、必ず一次資料（`references/` のシラバス）と突き合わせて構成・用語を確認する。

## 執筆スタイル

レッスンを書く・直すときは以下を守ります。

- **対象読者**: ソフトウェア開発の経験はあるが体系的なテストの知識は浅く、これから JSTQB FL 受験を目指す社会人・エンジニア。1 レッスン 15 分程度で読める分量。
- **文体**: です・ます体。
- **構成**: 各レッスンは次の順で書く。
  1. `# lessonNN: テーマ — サブタイトル`
  2. `## このレッスンで学ぶこと`（3〜5 個の箇条書き）
  3. 本文（`## セクション` / `### サブセクション`）
  4. `## キーワード`（用語と説明の表）
  5. `## 試験のポイント`（出題の切り口の箇条書き）
- **キーワードと試験のポイントは役割を分ける（本文の再掲を避ける）**: この2セクションは本文をなぞる場所ではない。次の役割に尖らせ、コンテンツ量が少ないレッスンでも「同じことをもう一度書く」状態にしない。
  - `## キーワード`: 用語と定義の対応表。**定義を一度確認しておきたい語だけ**に絞る（本文で説明しきった自明な語は載せない）。表記はシラバス（用語集）に合わせる。レッスンをまたぐ共通語は[用語集](/glossary/)が担うので、ここは当該レッスン固有の中核語に寄せる。
  - `## 試験のポイント`: 本文の要約ではなく、**受験時に効く「切り口」**を書く。各項目は次のいずれかの価値を持たせる。(1) 紛らわしい概念の**区別**（例: 確認テスト⇔リグレッションテスト、テストレベル⇔テストタイプ）、(2) **ひっかけ**になりやすい言い回しや例外、(3) **頻出**の学習目標、(4) その学習目標の **K レベル**（K3 なら「導出・計算が問われる」）、(5) K3 技法の**計算・手順**の勘所。単に本文の一文を言い換えただけの項目は削る。
- **図解**: 必要に応じて Mermaid・表・`::: tip` / `::: warning` / `::: info` ブロックを使う。
- **Mermaid のラベル内改行は `\n` を使う**（`<br/>` は使わない。混在させない）。
- **相互参照**は `[lessonNN](/lessons/lessonNN/)` 形式のリンクにする。
- **位置依存の参照を使わない**: 「次のレッスン」「前のレッスン」「前章」など位置を表す語でレッスンを指さない。必ず `[lessonNN]` リンクで指す（カリキュラム再編で壊れないため）。
- **長い段落を作らない**: 3 文以上続く説明や複数の論点を含む段落は、短文・箇条書き・表・図に分解する。「削って短くする」のではなく「分解して読みやすくする」。
- **ピラミッド構造**: テーマ → 柱（3 つ程度）→ 各柱の詳細、の順で設計する。
- **見出しは体言止め・名詞句**で簡潔に。「〜しよう」「〜してみる」のような話しかけ調にしない。
- **セクション区切りに `---`（水平線）を使わない**。見出し自体が区切りになる。
- **本文中に em dash（——）を使わない**（文を分けるかカッコを使う）。レッスンタイトルの `タイトル — サブタイトル` の区切りは例外。
- **アクセシビリティを意識**: 図には意味の伝わる `alt` を必ず付け、色だけに依存しない説明を心がける。

## 図版（SVG / Mermaid）のルール

- Mermaid は本文に ` ```mermaid ` で書く。複雑な構造の SVG 図は `docs/public/diagrams/*.svg` に置き `![alt](/diagrams/<name>.svg)` で参照。
- **ファイル名はトピック名**（例: `test-levels.svg`）。`lessonNN` を含めない（カリキュラム再編で破綻するため）。
- **`alt` は省略しない**。図の主旨が伝わる説明を書く。`alt="図"` のような無情報文字列は禁止。
- **ダークモード対応必須**: SVG 内に `<style>` を持たせ、`@media (prefers-color-scheme: dark)` で色を切り替える。白背景・黒文字を固定しない。
- **図は必ず画像へレンダリングして目視確認する**（色依存・重なり・はみ出し・ラベルの読みやすさ。座標の読みだけで判断しない）。グレースケールでも意味が区別できるように、色以外の手がかり（形・記号・ラベル）を併用する。
- 本文に素の `<script>` / `<style>` を書かない（VitePress は SPA でグローバルスコープを共有する）。ドリルページの `<script setup>` は可。

## 用語の統一ルール（重要）

教材全体で以下を統一します。シラバス V4.0 の表記を正とする。

- **エラー / 欠陥 / 故障 / 根本原因**: この 4 語を厳密に区別する。「エラー（誤り）＝人の行為」「欠陥（defect、バグ）＝成果物の不備」「故障（failure）＝実行時に現れる事象」。**「失敗」ではなく「不合格」、「不具合」ではなく「欠陥」**（J02 改訂の用語に合わせる）。
- **確認テスト / リグレッションテスト**: 「確認テスト」（修正の確認）と「リグレッションテスト」（デグレードの検出）を混同しない。
- **デシジョンテーブル**: 条件（condition）とアクション（action）を行に、各列を「ルール」とする。真偽は T/F 系で表す（Y/N と混同しない）。
- **状態遷移**: 状態・イベント・遷移・アクション（ガード条件）で構成する。
- **リスク**: 「リスクの可能性（likelihood）」×「リスクの影響（impact）」でリスクレベルを表す。「発生可能性／影響度」など独自語に置き換えない。
- **テストレベル / テストタイプ**を混同しない（レベル＝いつ・何を対象に、タイプ＝どの品質特性に着目）。
- **開発担当者**（開発者ではなく）、**メトリクス**（メトリックではなく）、**実例**（例示との使い分け）など J02 の訳語に従う。
- 略語は初出で正式名称を添える（例: 「ATDD（受け入れテスト駆動開発）」「SDLC（ソフトウェア開発ライフサイクル）」）。
- 二次情報（解説記事・検索要約）は入口にとどめ、用語の表記は必ずシラバス（一次情報）で確認する。

## ドリル（quiz）

- 問題データは `docs/quiz/data/chapterN.ts` に `Quiz[]` 型で記述する（`docs/quiz/types.ts` 参照）。
- `id` は全体で一意（`q101` のように「q + 章番号 + 連番」）。`lesson` は実在する `lessonNN` を指す。`choices` は 4 つ・重複なし。`answer` は 0〜3。正解位置は散らす。
- 設問の文末は「〜ですか。」に統一する（「？」は使わない）。
- 難易度（`difficulty`）は `easy` / `normal` / `hard` をバランスよく。**シラバスの K レベルに対応づける目安**: K1→easy / K2→normal / K3→hard。K3 の技法レッスンには、実際にテストケースを導出する・カバレッジを計算するタイプの適用問題を入れる。
- 問題・解説はレッスン本文の事実と一致させる。
- **解説で他の設問の答えを明かさない**: 誤答を退けるとき、その誤答（＝別の設問の正解になりうる用語）の定義まで列挙しない。概念名で軽く退けるにとどめる。
- 章別ドリル（`chapterN/index.md`）は**固定順**で出題する（`:shuffle` は付けない）。「途中から再開」を成立させるため。ランダム出題はランダム系ページ（`random` / `random-5` / `random-10` / `random-40`）が担う。
- **選択肢の表示順は `QuizCard.vue` がクライアントで Fisher-Yates シャッフルする。SSR では identity 順で描画し、`onMounted` でシャッフルする**こと。回答の保存は常に元 index で行う。
- **ドリルの `question` / `explanation` では `**`（太字）を使わない**。QuizCard の `renderText` はバッククォート（`` `code` ``）しか変換しない。
- 追加・編集後は必ず `npm run quiz:validate` で検証する。

## 日本語品質チェックの多層構成

1. **`natural-japanese` スキル**（アカウント標準）— 直訳調・詩的常套句・説明の足場・文中ダッシュ・長すぎる段落を検出。全レッスン下書きと主要ドキュメントに適用。
2. **`review-lesson` スキル**（`.claude/skills/`）— A〜I の全カテゴリを指摘ゼロまでループ。技術的正確性（シラバス一次照合）＋構成＋自然な日本語＋ビルド健全性。この中から `natural-japanese` も呼ぶ。
3. **機械フック**（`.claude/hooks/`）— `check-bold.js`（`**` 崩れ）、`check-style.js`（複文・詩的見出し・全角記号・足場）、`check-lesson-title.js`（h1 形式）、`textlint`、`quiz-validate`。

## Markdown の太字（強調）について

日本語では `**強調**` の閉じ `**` の直前が全角閉じ括弧（`）」】〉》`）だと、素の markdown-it では強調が描画されない（CJK flanking 問題）。このプロジェクトは `markdown-it-cjk-friendly` を導入して解消済みのため、本文では `**用語（よみ）**` のような書き方をしてもよい。ただし**ドリル（quiz）では `**` を使わない**。

## よく使うコマンド

```bash
npm run docs:dev       # ローカル開発サーバー
npm run docs:build     # 本番ビルド（Vercel と同じ）
npm run docs:lint      # textlint で日本語を校正
npm run quiz:validate  # ドリルデータの検証
npm run pwa:icons      # logo.svg から PWA アイコンを生成
```

## 変更時のチェック

レッスンやドリルを変更したら、コミット前に最低限これらを通す。

- `npm run docs:lint` がエラーなしで通る
- `npm run docs:build` が成功する
- ドリルを触ったら `npm run quiz:validate` も通る
- レッスンを触ったら `node .claude/hooks/check-bold.js docs/lessons/*/index.md` と `node .claude/hooks/check-style.js` の出力が空
